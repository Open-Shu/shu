🌱 Startup Seeding
1
Operator sets env vars

SHU_STRIPE_CUSTOMER_ID and SHU_STRIPE_SUBSCRIPTION_ID are set at deploy time for this instance's tenant.

2
ensure_singleton() on startup

Creates billing_state row (id=1) if absent. Safe under concurrent multi-worker startup via SAVEPOINT.

3
seed_from_config() writes NULL fields only

Writes customer_id + subscription_id from env into billing_state. Never overwrites values already set by webhooks or a previous seed.

📊 Usage Metering (Hourly)
1
UsageReportingSource fires

SchedulerService calls cleanup_stale() every hour (configurable via SHU_STRIPE_USAGE_REPORT_INTERVAL).

2
Query our total

UsageProviderImpl aggregates llm_usage.total_cost for the current billing period → microdollars.

3
Query Stripe's total

get_meter_event_summary() fetches Stripe's aggregated view for the same window.

4
Compare-and-correct

delta = our_total − max(stripe_total, last_reported_total). Async-lag protection prevents double-counting.

5
Report delta

MeterEvent sent to Stripe. last_reported_total + period saved to billing_state for next run.

🛡️ Multi-Instance Safety

Multiple Shu instances can share one Stripe account. Each instance only processes events for its own tenant:


  • expected_customer_id guard — if SHU_STRIPE_CUSTOMER_ID is not configured, all webhook events are dropped with a warning. Once configured, events for other customers are silently rejected.
  • Operator-provisioned identity — the Stripe customer and subscription IDs are set by the operator at deploy time; Shu never creates them itself.
  • Row-level locking — billing_state mutations use SELECT ... FOR UPDATE so concurrent webhook handlers serialise at the DB level without lost updates.
